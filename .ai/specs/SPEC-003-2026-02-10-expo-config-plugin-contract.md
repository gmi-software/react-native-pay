# SPEC-003: Expo Config Plugin Contract

## Overview

This specification documents the behavior of the Expo config plugin exported by the package, including how it configures Apple Pay entitlements and Android Google Pay manifest metadata during prebuild.

## Problem Statement

Payment runtime support depends on native project configuration. If plugin behavior is unclear, integrators may misconfigure app.json/app.config and end up with unavailable payment capabilities at runtime.

## Proposed Solution

Define a clear contract for plugin inputs and native modifications:

- Plugin entrypoint composes iOS and Android plugin steps.
- iOS step adds merchant identifiers to entitlements when configured.
- Android step toggles Google Pay metadata in main application manifest.

## Architecture

Source files:

- `package/src/plugin/index.ts` - plugin composition entrypoint
- `package/src/plugin/type.ts` - plugin props contract
- `package/src/plugin/withApplePay.ts` - iOS entitlement mutation
- `package/src/plugin/withGooglePay.ts` - Android manifest metadata mutation

Execution model:

1. Expo prebuild runs plugin with user props.
2. `withGooglePay` mutates Android manifest metadata.
3. `withApplePay` mutates iOS entitlements if merchant IDs provided.
4. Native project files generated by Expo include required capability flags.

## Data Models

Plugin props:

```ts
interface ReactNativePayPluginProps {
  merchantIdentifier?: string | string[]
  enableGooglePay?: boolean
}
```

Normalized behavior:

- `merchantIdentifier` accepts string or array; internal logic normalizes to array.
- Duplicate merchant IDs are not re-added.
- `enableGooglePay` defaults to `false`.

## API Contracts

### iOS Contract (`withApplePay`)

- If `merchantIdentifier` is missing, no entitlement mutation is performed.
- If provided, values are added to `com.apple.developer.in-app-payments`.
- Existing entitlement values are preserved and extended.

### Android Contract (`withGooglePay`)

- When `enableGooglePay=true`, add metadata:
  - name: `com.google.android.gms.wallet.api.enabled`
  - value: `true`
- When `enableGooglePay=false`, remove that metadata from main application.

## UI/UX

Not applicable (build-time configuration only).

## Configuration

Supported user config examples:

- Apple Pay only:
  - `{ merchantIdentifier: "merchant.com.example.app" }`
- Google Pay only:
  - `{ enableGooglePay: true }`
- Both:
  - `{ merchantIdentifier: "merchant.com.example.app", enableGooglePay: true }`

Expected consumer workflow:

1. Configure plugin in `app.json` or `app.config.*`.
2. Run `expo prebuild` (often with `--clean` when validating config changes).
3. Build/run native apps.

## Alternatives Considered

1. **Separate package plugins for iOS and Android**
   - Rejected to keep a single package integration point.
2. **Always set Google Pay metadata (no toggle)**
   - Rejected to avoid forcing config for apps that do not use Google Pay.
3. **Throw on missing merchant identifier**
   - Rejected because Android-only apps should be valid.

## Implementation Approach

When changing plugin behavior:

1. Update plugin source and type definitions.
2. Add/adjust plugin unit tests in `package/src/plugin/__tests__/`.
3. Update root `README.md` setup examples.
4. Document behavior updates in this spec changelog.

## Migration Path

No migration required for initial baseline documentation.

Future plugin contract changes must include:

- Backward compatibility impact
- Required app config updates
- Prebuild regeneration guidance

## Success Metrics

- Integrators can configure Apple Pay/Google Pay correctly on first setup.
- Plugin tests detect regressions in entitlement/manifest mutation logic.
- Support issues caused by config ambiguity decrease.

## Open Questions

- Should plugin validate merchant identifier format and warn early?
- Should Google Pay metadata support custom boolean value typing beyond string `"true"`?
